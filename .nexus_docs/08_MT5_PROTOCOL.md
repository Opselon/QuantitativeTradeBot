# 08_MT5_PROTOCOL – Nexus MT5 Bridge Architecture

## 1. Overview

This document specifies how the Nexus Trading Engine integrates with MetaTrader 5 (MT5) via a dedicated, real TCP bridge:

- **Nexus (.NET / C#)** is the *brain* (strategies, risk, portfolio, orchestration).
- **MetaTrader 5 Terminal** is the broker client.
- **`NexusBridge.mq5` Expert Advisor (EA)** inside MT5 is the *executor* that:
  - Receives commands from Nexus over a bridge (TCP/WebSocket/etc.).
  - Calls MT5 APIs to read account/positions/orders and (later) place trades.
  - Sends structured JSON responses back to Nexus.

The communication model:

```text
[Nexus C#]  <== JSON over TCP ==>  [NexusBridge.mq5 EA inside MT5]  <===> [MT5 Server/Broker]
```

The goal is to have a **production-ready MT5 connectivity path** that coexists with the existing **simulated MT5 adapter**, respecting clean architecture boundaries and DDD/CQRS principles.

## 3. Bridge Message Contracts (JSON)

All messages exchanged between Nexus and `NexusBridge.mq5` use a unified JSON envelope.

### 3.1. Envelope Structure

Logical (language-agnostic) format:

```json
{
  "messageType": "Request",   // or "Response"
  "requestId": "GUID-or-string",
  "command": "GetAccountSnapshot",  // or "Ping", "PlaceOrder", etc.
  "payload": { /* command-specific data */ },
  "error": {
    "code": "…",
    "message": "…"
  },
  "version": "1.0"
}
```

Fields:

- `messageType`  
  - `"Request"`: Nexus → MT5  
  - `"Response"`: MT5 → Nexus
- `requestId`  
  - Generated by Nexus (e.g., GUID).
  - Echoed by MQL5 in the response for correlation.
- `command`  
  - Logical command name (`GetAccountSnapshot`, `Ping`, `PlaceOrder`, etc.).
- `payload`  
  - Request or response data specific to the command.
- `error`  
  - `null` for success.
  - Object on failure: includes `code`, `message`, and optional extra details.

### 3.2. Example – `GetAccountSnapshotRequest`

Nexus → MT5:

```json
{
  "messageType": "Request",
  "requestId": "8bf3a207-7e5e-4eb2-a3c0-8f57ea123456",
  "command": "GetAccountSnapshot",
  "payload": null,
  "error": null,
  "version": "1.0"
}
```

### 3.3. Example – `GetAccountSnapshotResponse`

MT5 → Nexus:

```json
{
  "messageType": "Response",
  "requestId": "8bf3a207-7e5e-4eb2-a3c0-8f57ea123456",
  "command": "GetAccountSnapshot",
  "payload": {
    "accountId": 12345678,
    "broker": "MyBroker-Live",
    "currency": "USD",
    "balance": 10000.50,
    "equity": 10050.75,
    "margin": 500.00,
    "freeMargin": 9550.75,
    "leverage": 100,
    "connectionHealth": "Healthy"
  },
  "error": null,
  "version": "1.0"
}
```

Payload fields are populated via MT5 Account APIs:

- `accountId` → `AccountInfoInteger(ACCOUNT_LOGIN)`
- `broker` → `AccountInfoString(ACCOUNT_COMPANY)`
- `currency` → `AccountInfoString(ACCOUNT_CURRENCY)`
- `balance` → `AccountInfoDouble(ACCOUNT_BALANCE)`
- `equity` → `AccountInfoDouble(ACCOUNT_EQUITY)`
- `margin` → `AccountInfoDouble(ACCOUNT_MARGIN)`
- `freeMargin` → `AccountInfoDouble(ACCOUNT_FREEMARGIN)`
- `leverage` → `AccountInfoInteger(ACCOUNT_LEVERAGE)`

`connectionHealth`:

- `"Healthy"` if all data is retrieved without errors.
- `"Degraded"` if any API fails or returns inconsistent values.

## 4. MQL5 – `NexusBridge.mq5` Specification

The Expert Advisor `MQL5/Experts/Nexus/NexusBridge.mq5` implements a robust background stream processor over standard MT5 socket functionality.

### 4.1. Core Mechanics
- **Connection**: In `OnInit()`, the EA triggers `SocketCreate()` and `SocketConnect()` to connect to the C# endpoint running as a TCP Listener.
- **Heartbeat & Event Loop**: Scheduled using `EventSetMillisecondTimer()`. In `OnTimer()`, the EA reads stream buffers, checks connection status, and attempts automatic reconnection if offline.
- **Serialization**: Encodes messages using manual string concatenation to maintain high speed and prevent heavy library dependencies.
- **Routing**: Splits stream bytes on `\n` characters to parse individual discrete messages, using lightweight JSON key extractors to read the envelope and dispatch commands.

## 5. Mode Selection – Simulated vs Real Bridge

The C# engine supports dual-mode execution:

### 5.1. Configuration
```json
{
  "SelectedProvider": "PostgreSQL",
  "Mt5Mode": "Real",
  "Mt5BridgeHost": "127.0.0.1",
  "Mt5BridgePort": 5000,
  "Mt5BridgeUseSsl": false
}
```

### 5.2. Routing
DI registrations dynamically load `RoutingMt5ConnectionService` and `RoutingMt5AccountService`.
Calls are automatically delegated to the active adapter:
- `SimulatedMt5ConnectionService` / `SimulatedMt5AccountService` (for offline dev/evaluation).
- `RealMt5BridgeConnectionService` / `RealMt5BridgeAdapter` (for real bridge execution).
