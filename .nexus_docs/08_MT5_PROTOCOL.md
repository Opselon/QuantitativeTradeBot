# 08_MT5_PROTOCOL – Nexus MT5 Bridge Architecture

## 1. Overview

This document specifies how the Nexus Trading Engine integrates with MetaTrader 5 (MT5) via a dedicated bridge:

- **Nexus (.NET / C#)** is the *brain* (strategies, risk, portfolio, orchestration).
- **MetaTrader 5 Terminal** is the broker client.
- **`NexusBridge.mq5` Expert Advisor (EA)** inside MT5 is the *executor* that:
  - Receives commands from Nexus over a bridge (TCP/WebSocket/etc.).
  - Calls MT5 APIs to read account/positions/orders and (later) place trades.
  - Sends structured JSON responses back to Nexus.

The communication model:

```text
[Nexus C#]  <== JSON over TCP/WebSocket ==>  [NexusBridge.mq5 EA inside MT5]  <===> [MT5 Server/Broker]
```

The goal is to have a **production-ready MT5 connectivity path** that coexists with the existing **simulated MT5 adapter**, respecting clean architecture boundaries and DDD/CQRS principles.

## 3. Bridge Message Contracts (JSON)

All messages exchanged between Nexus and `NexusBridge.mq5` use a unified JSON envelope.

### 3.1. Envelope Structure

Logical (language-agnostic) format:

```json
{
  "messageType": "Request",   // or "Response"
  "requestId": "GUID-or-string",
  "command": "GetAccountSnapshot",  // or "Ping", "PlaceOrder", etc.
  "payload": { /* command-specific data */ },
  "error": {
    "code": "…",
    "message": "…"
  }
}
```

Fields:

- `messageType`  
  - `"Request"`: Nexus → MT5  
  - `"Response"`: MT5 → Nexus
- `requestId`  
  - Generated by Nexus (e.g., GUID).
  - Echoed by MQL5 in the response for correlation.
- `command`  
  - Logical command name (`GetAccountSnapshot`, `Ping`, `PlaceOrder`, etc.).
- `payload`  
  - Request or response data specific to the command.
- `error`  
  - `null` for success.
  - Object on failure: includes `code`, `message`, and optional extra details.

### 3.2. Example – `GetAccountSnapshotRequest`

Nexus → MT5:

```json
{
  "messageType": "Request",
  "requestId": "8bf3a207-7e5e-4eb2-a3c0-8f57ea123456",
  "command": "GetAccountSnapshot",
  "payload": {}
}
```

For now, no additional payload fields are required; MT5 reads the current account.

### 3.3. Example – `GetAccountSnapshotResponse`

MT5 → Nexus:

```json
{
  "messageType": "Response",
  "requestId": "8bf3a207-7e5e-4eb2-a3c0-8f57ea123456",
  "command": "GetAccountSnapshot",
  "payload": {
    "accountId": 12345678,
    "broker": "MyBroker-Live",
    "currency": "USD",
    "balance": 10000.50,
    "equity": 10050.75,
    "margin": 500.00,
    "freeMargin": 9550.75,
    "leverage": 100,
    "connectionHealth": "Healthy"
  },
  "error": null
}
```

Payload fields are populated via MT5 Account APIs:

- `accountId` → `AccountInfoInteger(ACCOUNT_LOGIN)`
- `broker` → `AccountInfoString(ACCOUNT_COMPANY)`
- `currency` → `AccountInfoString(ACCOUNT_CURRENCY)`
- `balance` → `AccountInfoDouble(ACCOUNT_BALANCE)`
- `equity` → `AccountInfoDouble(ACCOUNT_EQUITY)`
- `margin` → `AccountInfoDouble(ACCOUNT_MARGIN)`
- `freeMargin` → `AccountInfoDouble(ACCOUNT_FREEMARGIN)`
- `leverage` → `AccountInfoInteger(ACCOUNT_LEVERAGE)`

`connectionHealth`:

- `"Healthy"` if all data is retrieved without errors.
- `"Degraded"` or `"Unknown"` if any API fails or returns inconsistent values.

### 3.4. Example – `Ping`

Request:

```json
{
  "messageType": "Request",
  "requestId": "ping-001",
  "command": "Ping",
  "payload": {}
}
```

Response:

```json
{
  "messageType": "Response",
  "requestId": "ping-001",
  "command": "Ping",
  "payload": {
    "serverTime": "2025-01-01T12:34:56",
    "terminalTime": "2025-01-01T12:34:57",
    "status": "Alive"
  },
  "error": null
}
```

## 4. .NET Contracts & Ports

### 4.1. `BridgeMessageEnvelope.cs`

Location: `src/Nexus.Application/Mt5Bridge/Contracts/BridgeMessageEnvelope.cs`

Conceptual properties:

- `string MessageType`  // `"Request"` or `"Response"`
- `string RequestId`
- `string Command`
- `object? Payload`    // or generic `TPayload`
- `BridgeError? Error`

Design guidelines:

- Immutable wherever possible:
  - Constructors assign all fields, only getters exposed.
- JSON-friendly:
  - Use `System.Text.Json` with `[JsonPropertyName(...)]` if needed.
- Versionable:
  - Optionally include a `Version` field for future extension.

### 4.2. `BridgeError.cs`

Location: `src/Nexus.Application/Mt5Bridge/Contracts/BridgeError.cs`

Fields:

- `string Code`
- `string Message`
- `string? Details` (optional)

Logging:

- Should be safe to log without exposing secrets.
- Do not embed passwords, tokens, or raw credentials.

### 4.3. `GetAccountSnapshotRequest.cs`

Location: `src/Nexus.Application/Mt5Bridge/Contracts/GetAccountSnapshotRequest.cs`

Initially:

- Can be an empty DTO or include minimal fields for future use.
- Primarily acts as a typed marker.

### 4.4. `GetAccountSnapshotResponse.cs`

Location: `src/Nexus.Application/Mt5Bridge/Contracts/GetAccountSnapshotResponse.cs`

Fields:

- `long AccountId`
- `string Broker`
- `string Currency`
- `decimal Balance`
- `decimal Equity`
- `decimal Margin`
- `decimal FreeMargin`
- `int Leverage`
- `string ConnectionHealth`  
  - Should map to a domain `enum` (e.g., `ConnectionHealthStatus`) in the application layer.

### 4.5. `IMt5BridgeClient.cs`

Location: `src/Nexus.Application/Ports/IMt5BridgeClient.cs`

Responsibilities:

- Abstracts transport (TCP/WebSocket/etc.) behind a consistent API.
- No direct awareness of sockets, streams, or MT5 specifics.

Methods (conceptually):

- `Task ConnectAsync(CancellationToken ct)`
- `Task DisconnectAsync(CancellationToken ct)`
- `Task<BridgeMessageEnvelope> SendAsync(BridgeMessageEnvelope request, CancellationToken ct)`
- Optional:
  - `IAsyncEnumerable<BridgeMessageEnvelope> ListenAsync(CancellationToken ct)` for streaming notifications.

Characteristics:

- Pure application port:
  - Only deals with `BridgeMessageEnvelope` DTOs.
  - No direct UI or infrastructure dependencies.

### 4.6. `TcpMt5BridgeClient.cs` (or `WebSocketMt5BridgeClient.cs`)

Location: `src/Nexus.Infrastructure/Mt5Bridge/`

Responsibilities:

- Maintains a connection to the configured endpoint:
  - e.g., `Mt5Bridge:Host`, `Mt5Bridge:Port`, `Mt5Bridge:UseTls`.
- Serializes/deserializes JSON envelopes using `System.Text.Json`.
- Manages:
  - Request/response correlation (e.g., dictionary of `RequestId → TaskCompletionSource<BridgeMessageEnvelope>`).
  - Timeouts (e.g., fail with `BridgeError` `"Timeout"`).
  - Reconnect behavior upon failure.
  - Structured logging (metadata only).

Configuration:

- Uses existing configuration/app configuration service.
- No hard-coded host/port.
- Sensitive values (like credentials, if any later) must be protected via existing secret store (`ISecretStore` / `WindowsSecretStore`).

## 5. MQL5 – `NexusBridge.mq5` Specification

This section specifies how the `NexusBridge.mq5` EA should be implemented so that a human MQL5 developer can build it correctly.

### 5.1. File Header

At the top of `MQL5/Experts/Nexus/NexusBridge.mq5`:

```c++
#property copyright "Nexus Trading Engine"
#property link      "https://nexus.example.com"
#property version   "1.0"
#property strict

// Optional description:
// #property description "Nexus MT5 Bridge EA - connects MT5 terminal to Nexus backend via JSON messages."
```

### 5.2. Global Variables

Examples:

```c++
int   bridge_socket = INVALID_HANDLE;  // or appropriate type for network handle
bool  is_connected  = false;

string bridge_host = "127.0.0.1";
int    bridge_port = 5000;

// Buffers for message handling
string incoming_buffer = "";
string outgoing_buffer = "";
```

Notes:

- Actual socket implementation depends on MT5 environment capabilities:
  - Direct sockets (if supported).
  - Wrapper DLL.
  - File-based IPC (fallback).
- The protocol-level logic (JSON envelopes, commands, payloads) must remain as specified here.

### 5.3. `OnInit()`

Responsibilities:

1. Load configuration (bridge host, port, etc.).
2. Establish connection to Nexus bridge.
3. Set up a timer for periodic polling.

Pseudo-code:

```c++
int OnInit()
{
   // TODO: load configuration from file or inputs
   // e.g., FileOpen("NexusBridge.cfg", ...), read bridge_host/bridge_port

   if(!ConnectToBridge(bridge_host, bridge_port))
   {
      Print("NexusBridge: failed to connect to bridge at ", bridge_host, ":", bridge_port);
      return(INIT_FAILED);
   }

   is_connected = true;

   // Poll messages every 1 second (adjust if needed)
   EventSetTimer(1);

   Print("NexusBridge: initialized and connected.");
   return(INIT_SUCCEEDED);
}
```

### 5.4. `OnDeinit(const int reason)`

Responsibilities:

- Kill timer.
- Close connection.

Pseudo-code:

```c++
void OnDeinit(const int reason)
{
   EventKillTimer();

   if(is_connected)
   {
      DisconnectFromBridge();
      is_connected = false;
   }

   Print("NexusBridge: deinitialized, connection closed.");
}
```

### 5.5. `OnTimer()` – Message Loop

Prefer using `OnTimer()` for polling:

```c++
void OnTimer()
{
   if(!is_connected)
   {
      AttemptReconnect();
      return;
   }

   string msg;
   while(ReadNextMessage(msg))
   {
      ProcessIncomingMessage(msg);
   }

   // Optional: SendHeartbeatIfNeeded();
}
```

Functions to be implemented:

- `AttemptReconnect()`
- `ReadNextMessage(string &msg)`
- `ProcessIncomingMessage(const string msg)`

### 5.6. `ProcessIncomingMessage`

Responsibilities:

- Parse JSON envelope.
- Dispatch based on `command`.

Pseudo-code:

```c++
void ProcessIncomingMessage(const string msg)
{
   string messageType;
   string requestId;
   string command;
   string payloadJson;

   if(!ParseEnvelope(msg, messageType, requestId, command, payloadJson))
   {
      Print("NexusBridge: failed to parse message envelope: ", msg);
      // Optionally send an error response back
      return;
   }

   if(messageType != "Request")
   {
      Print("NexusBridge: ignoring non-request messageType: ", messageType);
      return;
   }

   if(command == "GetAccountSnapshot")
   {
      HandleGetAccountSnapshot(requestId, payloadJson);
   }
   else if(command == "Ping")
   {
      HandlePing(requestId, payloadJson);
   }
   else
   {
      HandleUnknownCommand(requestId, command);
   }
}
```

### 5.7. `HandleGetAccountSnapshot`

Reads account information via MT5 APIs and sends a response.

Pseudo-code:

```c++
void HandleGetAccountSnapshot(const string requestId, const string payloadJson)
{
   long   accountId   = AccountInfoInteger(ACCOUNT_LOGIN);
   string broker      = AccountInfoString(ACCOUNT_COMPANY);
   string currency    = AccountInfoString(ACCOUNT_CURRENCY);
   double balance     = AccountInfoDouble(ACCOUNT_BALANCE);
   double equity      = AccountInfoDouble(ACCOUNT_EQUITY);
   double margin      = AccountInfoDouble(ACCOUNT_MARGIN);
   double freeMargin  = AccountInfoDouble(ACCOUNT_FREEMARGIN);
   int    leverage    = (int)AccountInfoInteger(ACCOUNT_LEVERAGE);

   string connectionHealth = "Healthy";
   // TODO: adjust connectionHealth if any API call fails or returns invalid data

   string payloadResp = "{"
      "\"accountId\":"        + IntegerToString(accountId) + ","
      "\"broker\":\""         + broker                     + "\","
      "\"currency\":\""       + currency                   + "\","
      "\"balance\":"          + DoubleToString(balance, 2) + ","
      "\"equity\":"           + DoubleToString(equity, 2)  + ","
      "\"margin\":"           + DoubleToString(margin, 2)  + ","
      "\"freeMargin\":"       + DoubleToString(freeMargin, 2) + ","
      "\"leverage\":"         + IntegerToString(leverage)  + ","
      "\"connectionHealth\":\"" + connectionHealth        + "\""
   "}";

   string responseJson = BuildResponseEnvelope(
      "Response",
      requestId,
      "GetAccountSnapshot",
      payloadResp,
      ""   // errorJson = "" ⇒ null
   );

   SendMessageToBridge(responseJson);
}
```

### 5.8. `HandlePing`

Responds to health checks / heartbeat.

Pseudo-code:

```c++
void HandlePing(const string requestId, const string payloadJson)
{
   datetime now = TimeCurrent();
   string   nowStr = TimeToString(now, TIME_DATE | TIME_SECONDS);

   string payloadResp = "{"
      "\"serverTime\":\""   + nowStr + "\","
      "\"terminalTime\":\"" + nowStr + "\","
      "\"status\":\"Alive\""
   "}";

   string responseJson = BuildResponseEnvelope(
      "Response",
      requestId,
      "Ping",
      payloadResp,
      ""
   );

   SendMessageToBridge(responseJson);
}
```

### 5.9. `HandleUnknownCommand`

Sends back a structured error response.

Pseudo-code:

```c++
void HandleUnknownCommand(const string requestId, const string command)
{
   string errorJson = "{"
      "\"code\":\"UnknownCommand\","
      "\"message\":\"Unsupported command: " + command + "\""
   "}";

   string responseJson = BuildResponseEnvelope(
      "Response",
      requestId,
      command,
      "{}",      // empty payload
      errorJson
   );

   SendMessageToBridge(responseJson);
}
```

### 5.10. JSON Envelope Builder

Constructs the JSON envelope from fields.

```c++
string BuildResponseEnvelope(
   const string messageType,
   const string requestId,
   const string command,
   const string payloadJson,
   const string errorJson // "" means null
)
{
   string json = "{"
      "\"messageType\":\"" + messageType + "\","
      "\"requestId\":\""   + requestId   + "\","
      "\"command\":\""     + command     + "\","
      "\"payload\":"       + payloadJson + ",";

   if(errorJson == "" || StringLen(errorJson) == 0)
      json += "\"error\":null";
   else
      json += "\"error\":" + errorJson;

   json += "}";

   return json;
}
```

On the Nexus side, `System.Text.Json` deserializes this into `BridgeMessageEnvelope` and corresponding DTOs.

## 6. Mode Selection – Simulated vs Real Bridge

Nexus must support two MT5 modes:

- **Simulated MT5**  
  - Current adapter providing synthetic data.
- **Real MT5 Bridge**  
  - Uses `IMt5BridgeClient` to talk to `NexusBridge.mq5`.

### 6.1. Configuration

Example configuration (e.g., `appsettings.json`):

```json
{
  "Mt5": {
    "Mode": "Simulated",       // or "Bridge"
    "Bridge": {
      "Host": "127.0.0.1",
      "Port": 5000,
      "UseTls": false
    }
  }
}
```

Desktop UI:

- MT5 Onboarding step:
  - A selection: `Mode = Simulated | Real Bridge`.
  - When `Real Bridge` is chosen:
    - Fields for `Host` and `Port` become active/editable.
- `AppConfigurationService` persists these settings.

### 6.2. Dependency Wiring

In the DI composition root:

- If `Mode == "Simulated"`:
  - `IMt5AccountService` → `SimulatedMt5Adapter`.
- If `Mode == "Bridge"`:
  - `IMt5AccountService` → `RealMt5BridgeAdapter` (which uses `IMt5BridgeClient`).

UI:

- Must clearly show active MT5 mode:
  - `Simulated`: explicitly marked as development/test.
  - `Real Bridge`: clearly labeled as connected to MT5 terminal via bridge.

## 7. Security & Logging

### 7.1. Security

- Bridge configuration (`Host`, `Port`, credentials if any in future) must be handled via:
  - Existing options/configuration abstractions.
  - Existing secret store (`ISecretStore`, `WindowsSecretStore`) where appropriate.
- Sensitive data:
  - **Never** logged:
    - Passwords
    - Tokens
    - Secret keys
    - Raw credentials or full payloads containing sensitive fields.

### 7.2. Logging / Observability

Nexus side:

- Log metadata only:
  - `RequestId`
  - `Command`
  - Status (`Success`, `Failed`, `Timeout`, `Degraded`, etc.).
  - High-level error codes.
- Diagnostics to capture:
  - Bridge startup.
  - Handshake success/failure.
  - Timeouts.
  - Reconnect attempts and outcomes.
  - Degraded or unknown health states.
  - Invalid payloads / parse failures.

MT5 side:

- Use `Print()` sparingly:
  - Log high-level issues (connection failure, parse errors).
  - Do not log full payloads with potentially sensitive data.

## 8. Testing & Local Validation

### 8.1. Nexus Tests

Recommended test coverage:

- **Serialization/Deserialization**
  - Given a `BridgeMessageEnvelope` with `GetAccountSnapshotResponse`, ensure JSON round-trip preserves fields.
- **Mapping**
  - `GetAccountSnapshotResponse` → `AccountSnapshotDto` used by the application layer.
- **Timeout Handling**
  - When a request gets no response within the configured timeout:
    - A `BridgeError` with code `"Timeout"` is produced.
- **Mode Selection**
  - When config is `"Simulated"`:
    - Simulated adapter is used, real bridge is not instantiated.
  - When config is `"Bridge"`:
    - `RealMt5BridgeAdapter` + `IMt5BridgeClient` are used.

### 8.2. Local Testing without Real MT5

Even without MT5 installed:

- Create a **fake local bridge server**:
  - Simple TCP listener that accepts connections from `TcpMt5BridgeClient`.
  - When it receives a JSON request:
    - It responds with a valid JSON envelope (e.g., synthetic `GetAccountSnapshotResponse`).
- Use this server to validate:
  - Connection logic.
  - Request/response correlation.
  - Timeout and reconnect behavior.

## 9. Future Extensions – Trade Execution

This document focuses on account snapshots and basic health checks. For future phases:

- Extend contracts with:
  - `PlaceOrderRequest`, `PlaceOrderResponse`
  - `ClosePositionRequest`, `ClosePositionResponse`
  - `ModifyOrderRequest`, etc.
- Each new command must:
  - Follow the same envelope pattern.
  - Be versioned.
  - Be safe to log (metadata-only).
- MQL5 must implement corresponding handlers using:
  - `OrderSend()`, `OrderClose()`, `OrderModify()`, etc.
- Nexus:
  - Keeps all strategy and risk logic.
  - Uses MT5 only as execution endpoint.

## 10. Summary

- **Nexus** orchestrates strategies and uses MT5 via a **bridge**.
- **`NexusBridge.mq5` EA** inside MT5:
  - Listens for JSON commands.
  - Executes MT5 APIs.
  - Returns JSON responses.
- Contracts:
  - Use a shared `BridgeMessageEnvelope` with `messageType`, `requestId`, `command`, `payload`, and `error`.
- Modes:
  - `Simulated` (existing path).
  - `Real Bridge` (new production path).
- Architecture:
  - Clean, DDD-compliant.
  - No direct UI ↔ infrastructure coupling.
  - Testable locally with or without MT5.

This document serves as the foundation for implementing and maintaining the MT5 bridge in the Nexus Trading Engine.
