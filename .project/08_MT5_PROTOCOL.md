# 08_MT5_PROTOCOL – Nexus MT5 Bridge Architecture

## 1. Overview

This document specifies how the Nexus Trading Engine integrates with MetaTrader 5 (MT5) via a dedicated, real TCP bridge:

- **Nexus (.NET / C#)** is the *brain* (strategies, risk, portfolio, orchestration).
- **MetaTrader 5 Terminal** is the broker client.
- **`NexusBridge.mq5` Expert Advisor (EA)** inside MT5 is the *executor* that:
  - Receives commands from Nexus over a bridge (TCP/WebSocket/etc.).
  - Calls MT5 APIs to read account/positions/orders and place trades.
  - Sends structured JSON responses back to Nexus.

The communication model:

```text
[Nexus C#]  <== JSON over TCP ==>  [NexusBridge.mq5 EA inside MT5]  <===> [MT5 Server/Broker]
```

The goal is to have a **production-ready MT5 connectivity path** that coexists with the existing **simulated MT5 adapter**, respecting clean architecture boundaries and DDD/CQRS principles.

## 3. Bridge Message Contracts (JSON)

All messages exchanged between Nexus and `NexusBridge.mq5` use a unified JSON envelope.

### 3.1. Envelope Structure

Logical (language-agnostic) format:

```json
{
  "messageType": "Request",   // or "Response"
  "requestId": "GUID-or-string",
  "command": "GetAccountSnapshot",  // or "Ping", "PlaceOrder", "ClosePosition", "GetOpenPositions"
  "payload": { /* command-specific data */ },
  "error": {
    "code": "…",
    "message": "…"
  },
  "version": "1.0"
}
```

Fields:

- `messageType`
  - `"Request"`: Nexus → MT5
  - `"Response"`: MT5 → Nexus
- `requestId`
  - Generated by Nexus (e.g., GUID).
  - Echoed by MQL5 in the response for correlation.
- `command`
  - Logical command name (`GetAccountSnapshot`, `Ping`, `PlaceOrder`, `ClosePosition`, `GetOpenPositions`).
- `payload`
  - Request or response data specific to the command.
- `error`
  - `null` for success.
  - Object on failure: includes `code`, `message`, and optional extra details.

### 3.2. Example – `GetAccountSnapshot`

Request:

```json
{
  "messageType": "Request",
  "requestId": "8bf3a207-7e5e-4eb2-a3c0-8f57ea123456",
  "command": "GetAccountSnapshot",
  "payload": null,
  "error": null,
  "version": "1.0"
}
```

Response:

```json
{
  "messageType": "Response",
  "requestId": "8bf3a207-7e5e-4eb2-a3c0-8f57ea123456",
  "command": "GetAccountSnapshot",

}
```

## 4. New Core Trading Protocols

These trading protocols are fully implemented and integrated across both C# (.NET) and MQL5 (MT5 Terminal) sides:
- **C# Port Method**: `IMt5TradingService` sends JSON TCP requests via `TcpMt5BridgeClient`.
- **MQL5 Terminal Side**: `NexusBridge.mq5` parses incoming commands, executes them with robustness safeguards (normalizations, filling mode fallbacks), and replies with structured responses.

### 4.1. `PlaceOrder` Command

Launches a market execution order inside the broker terminal.

- **C# Port Method**: `IMt5TradingService.PlaceMarketOrderAsync(symbol, side, volume, stopLoss, takeProfit, comment, clientCorrelationId, cancellationToken)`
- **C# Request DTO Shape**: `PlaceOrderRequest`
- **C# Response DTO Shape**: `PlaceOrderResponse` (mapped to application-level `PlaceOrderResult`)

#### 4.1.1. Request Payload

```json
{
  "messageType": "Request",
  "requestId": "req-999-place",
  "command": "PlaceOrder",
  "payload": {
    "symbol": "EURUSD",
    "side": "Buy",          // "Buy" or "Sell"
    "volume": 0.10,
    "stopLoss": 1.08000,    // Optional SL price
    "takeProfit": 1.10000,  // Optional TP price
    "comment": "Algo Order 1",
    "clientCorrelationId": "strat-abc-001"
  },
  "error": null,
  "version": "1.0"
}
```

#### 4.1.2. Successful Response Payload

```json
{
  "messageType": "Response",
  "requestId": "req-999-place",
  "command": "PlaceOrder",
  "payload": {
    "success": true,
    "ticket": 58923412,
    "status": "Executed",
    "brokerMessage": "Trade completed successfully.",
    "comment": "Algo Order 1"
  },
  "error": null,
  "version": "1.0"
}
```

#### 4.1.3. Failed Response Payload

```json
{
  "messageType": "Response",
  "requestId": "req-999-place",
  "command": "PlaceOrder",
  "payload": {
    "success": false,
    "ticket": 0,
    "status": "Failed",
    "brokerMessage": "OrderSend failed. Retcode: 10013, Error: Invalid Volume.",
    "comment": ""
  },
  "error": {
    "code": "TRADE_REJECTED",
    "message": "OrderSend failed. Retcode: 10013, Error: Invalid Volume."
  },
  "version": "1.0"
}
```

### 4.2. `ClosePosition` Command

Requests a closure (or partial close) of a designated open position ticket.

- **C# Port Method**: `IMt5TradingService.ClosePositionAsync(positionTicket, symbol, volume, cancellationToken)`
- **C# Request DTO Shape**: `ClosePositionRequest`
- **C# Response DTO Shape**: `ClosePositionResponse` (mapped to application-level `ClosePositionResult`)

*Note: Following architectural guidelines, passing the symbol directly eliminates the need for sequential nested network round-trips over the bridge to query the position’s symbol, maximizing pipeline performance.*

```json

  "requestId": "req-999-close",
  "command": "ClosePosition",
  "payload": {
    "ticket": 58923412,
    "symbol": "EURUSD",
    "volume": null          // null for full position closure
  },
  "error": null,
  "version": "1.0"
}
```

#### 4.2.2. Successful Response Payload

```json

    "success": true,
```

## 5. MQL5 – `NexusBridge.mq5` Handler Responsibilities

1. **JSON Property Extraction**: Features robust extraction helper methods (`GetJsonStringValue`, `GetJsonDoubleValue`, and `GetJsonIntValue`) capable of reading both quoted strings and unquoted numeric values inside payload trees.
2. **Trade API Validation**: Assures essential properties (Symbol, Volume, and Type) are fully populated and selectable on-terminal.
3. **MqlTradeRequest Execution**: Prepares trading requests matching MT5 execution filling policies and dispatches via native `OrderSend()`.
4. **Robustness and Normalizations (Stage 2 improvements)**:
   - **Dynamic Volume step rounding**: Rounds requested trade lots to the broker's specific `SYMBOL_VOLUME_STEP` and forces them between `SYMBOL_VOLUME_MIN` and `SYMBOL_VOLUME_MAX`.
   - **Dynamic Filling Mode Fallbacks**: Detects supported filling modes (`FOK` vs `IOC` vs `RETURN`) on the specific symbol dynamically to avoid filling-mode rejection codes (e.g., retcode 10030).
   - **Precision Pricing Fallbacks**: Resolves latest symbol ask/bid prices via native high-precision `SymbolInfoTick` triggers, falling back cleanly to `SymbolInfoDouble` if necessary. Normalizes trade stop-loss and take-profit prices based on symbol's exact decimal digits (`SYMBOL_DIGITS`).
5. **Structured Error Dispatching**: Maps terminal rejections to standardized bridge error codes (such as `TRADE_REJECTED`, `INVALID_PAYLOAD`, and `POSITION_NOT_FOUND`).
6. **Escape Serialization Safety**: Safely escapes string outputs to prevent malformed or broken JSON transfers.

## 7. Mode Selection – Simulated vs Real Bridge

The C# engine supports dual-mode execution:

### 7.1. Configuration
```json
{
  "SelectedProvider": "PostgreSQL",
  "Mt5Mode": "Real",
  "Mt5BridgeHost": "127.0.0.1",
  "Mt5BridgePort": 5000,
  "Mt5BridgeUseSsl": false
}
```

### 7.2. Routing
DI registrations dynamically load `RoutingMt5ConnectionService`, `RoutingMt5AccountService`, and `RoutingMt5TradeService`.
Calls are automatically delegated to the active adapter:
- `SimulatedMt5ConnectionService` / `SimulatedMt5AccountService` / `SimulatedMt5TradeService` (for offline dev/evaluation).
- `RealMt5BridgeConnectionService` / `RealMt5BridgeAdapter` (for real bridge execution).

## 9. UI Integration and Message Flows

With Stage 3, the manual trading operator dashboard (`Mt5TradingPanel.xaml`) has been fully integrated into the Desktop Workstation.

### 9.1. Architectural Layout
The UI is strictly separated from any physical infrastructure:
```text
[Mt5TradingPanel View (XAML)]
             │
             ▼
[Mt5TradingViewModel]
             │
             ▼
[IMt5OperatorService (Facade)]
             │
             ▼
[IMt5TradingService (Abstraction)] ──► [RoutingMt5TradingService]
                                                 │
                        ┌────────────────────────┴────────────────────────┐
                        ▼                                                 ▼
          [SimulatedMt5TradingService]                      [RealMt5TradingService]
```

### 9.2. End-to-End Message Flows

#### 1. Retrieve & Refresh Positions
- **Trigger**: Manual operator clicks "REFRESH POSITIONS" or the auto-refresh background loop timer fires (every 5 seconds).
- **Flow**:
  1. `Mt5TradingViewModel` calls `_operatorService.GetPositionsAsync()`.
  2. `Mt5OperatorService` invokes `_tradingService.GetOpenPositionsAsync()`.
  3. `RoutingMt5TradingService` resolves the active service:
     - **Simulated Mode**: Fetches in-memory simulated positions from `SimulatedMt5TradeService`.
     - **Real Mode**: Sends a `GetOpenPositions` JSON Request Envelope over TCP via `TcpMt5BridgeClient` to `NexusBridge.mq5`.
  4. The response flows back, gets converted into clean `DesktopPositionDto` items, and wraps in `DesktopPositionViewModel` to bind to the UI Grid on the main Dispatcher Thread.

#### 2. Execute Market Order (Buy/Sell)
- **Trigger**: Operator validates Symbol & Volume, then clicks "BUY MARKET" or "SELL MARKET".
- **Flow**:
  1. `Mt5TradingViewModel` executes `PlaceOrderAsync(symbol, side, volume)`.
  2. `Mt5OperatorService` maps side to `BridgeOrderSide` and calls `_tradingService.PlaceMarketOrderAsync()`.
  3. Active trade service compiles request and returns trade results (`Ticket`, `Success`, `ErrorMessage`).
  4. Operator service normalizes exceptions (e.g., `SocketException` mapped to "Connection lost") and returns `DesktopTradeResult`.
  5. UI updates StatusMessage/ErrorMessage and triggers an automated positional refresh.

#### 3. Close Active Position
- **Trigger**: Operator selects a row in the Grid and clicks "CLOSE POSITION".
- **Flow**:
  1. `Mt5TradingViewModel` dispatches `ClosePositionAsync(ticket, symbol)`.
  2. `Mt5OperatorService` invokes `_tradingService.ClosePositionAsync()`.
  3. Active implementation instructs close execution (on simulated positions or real terminal tickets).
  4. Results are processed, errors are normalized, and UI refreshes the active positions table.
