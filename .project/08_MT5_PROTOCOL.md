# 08_MT5_PROTOCOL – Nexus MT5 Bridge Architecture

## 1. Overview

This document specifies how the Nexus Trading Engine integrates with MetaTrader 5 (MT5) via a dedicated, real TCP bridge:

- **Nexus (.NET / C#)** is the *brain* (strategies, risk, portfolio, orchestration).
- **MetaTrader 5 Terminal** is the broker client.
- **`NexusBridge.mq5` Expert Advisor (EA)** inside MT5 is the *executor* that:
  - Receives commands from Nexus over a bridge (TCP/WebSocket/etc.).
  - Calls MT5 APIs to read account/positions/orders and place trades.
  - Sends structured JSON responses back to Nexus.

The communication model:

```text
[Nexus C#]  <== JSON over TCP ==>  [NexusBridge.mq5 EA inside MT5]  <===> [MT5 Server/Broker]
```

The goal is to have a **production-ready MT5 connectivity path** that coexists with the existing **simulated MT5 adapter**, respecting clean architecture boundaries and DDD/CQRS principles.

## 3. Bridge Message Contracts (JSON)

All messages exchanged between Nexus and `NexusBridge.mq5` use a unified JSON envelope.

### 3.1. Envelope Structure

Logical (language-agnostic) format:

```json
{
  "messageType": "Request",   // or "Response"
  "requestId": "GUID-or-string",
  "command": "GetAccountSnapshot",  // or "Ping", "PlaceOrder", "ClosePosition", "GetOpenPositions"
  "payload": { /* command-specific data */ },
  "error": {
    "code": "…",
    "message": "…"
  },
  "version": "1.0"
}
```

Fields:

- `messageType`
  - `"Request"`: Nexus → MT5
  - `"Response"`: MT5 → Nexus
- `requestId`
  - Generated by Nexus (e.g., GUID).
  - Echoed by MQL5 in the response for correlation.
- `command`
  - Logical command name (`GetAccountSnapshot`, `Ping`, `PlaceOrder`, `ClosePosition`, `GetOpenPositions`).
- `payload`
  - Request or response data specific to the command.
- `error`
  - `null` for success.
  - Object on failure: includes `code`, `message`, and optional extra details.

### 3.2. Example – `GetAccountSnapshot`

Request:

```json
{
  "messageType": "Request",
  "requestId": "8bf3a207-7e5e-4eb2-a3c0-8f57ea123456",
  "command": "GetAccountSnapshot",
  "payload": null,
  "error": null,

}
```

Response:

```json

  "messageType": "Response",
  "requestId": "8bf3a207-7e5e-4eb2-a3c0-8f57ea123456",
  "command": "GetAccountSnapshot",
```

## 4. New Core Trading Protocols

In Stage 1, these trading protocols have been fully implemented and tested on the C# .NET side (Application & Infrastructure layers). The MT5 terminal/MQL5 Expert Advisor (`NexusBridge.mq5`) side of these commands is **planned for Stage 2** and does not yet exist (currently, only `Ping` and `GetAccountSnapshot` commands are fully integrated end-to-end with active MQL5 execution handlers).

### 4.1. `PlaceOrder` Command

Launches a market execution order inside the broker terminal.

- **C# Port Method**: `IMt5TradingService.PlaceMarketOrderAsync(symbol, side, volume, stopLoss, takeProfit, comment, clientCorrelationId, cancellationToken)`
- **C# Request DTO Shape**: `PlaceOrderRequest`
- **C# Response DTO Shape**: `PlaceOrderResponse` (mapped to application-level `PlaceOrderResult`)

#### 4.1.1. Request Payload

```json
{
  "messageType": "Request",
  "requestId": "req-999-place",
  "command": "PlaceOrder",
  "payload": {
    "symbol": "EURUSD",
    "side": "Buy",          // "Buy" or "Sell"
    "volume": 0.10,
    "stopLoss": 1.08000,    // Optional SL price
    "takeProfit": 1.10000,  // Optional TP price
    "comment": "Algo Order 1",
    "clientCorrelationId": "strat-abc-001"
  },
  "error": null,
  "version": "1.0"
}
```

#### 4.1.2. Successful Response Payload

```json
{
  "messageType": "Response",
  "requestId": "req-999-place",
  "command": "PlaceOrder",
  "payload": {
    "success": true,
    "ticket": 58923412,
    "status": "Executed",
    "brokerMessage": "Trade completed successfully.",
    "comment": "Algo Order 1"
  },
  "error": null,
  "version": "1.0"
}
```

#### 4.1.3. Failed Response Payload

```json
{
  "messageType": "Response",
  "requestId": "req-999-place",
  "command": "PlaceOrder",
  "payload": {
    "success": false,
    "ticket": 0,
    "status": "Failed",
    "brokerMessage": "OrderSend failed. Retcode: 10013, Error: Invalid Volume.",
    "comment": ""
  },
  "error": {
    "code": "TRADE_REJECTED",
    "message": "OrderSend failed. Retcode: 10013, Error: Invalid Volume."
  },
  "version": "1.0"
}
```

### 4.2. `ClosePosition` Command

Requests a closure (or partial close) of a designated open position ticket.

- **C# Port Method**: `IMt5TradingService.ClosePositionAsync(positionTicket, symbol, volume, cancellationToken)`
- **C# Request DTO Shape**: `ClosePositionRequest`
- **C# Response DTO Shape**: `ClosePositionResponse` (mapped to application-level `ClosePositionResult`)

*Note: Following architectural guidelines, passing the symbol directly eliminates the need for sequential nested network round-trips over the bridge to query the position’s symbol, maximizing pipeline performance.*

#### 4.2.1. Request Payload

```json
{
  "messageType": "Request",
  "requestId": "req-999-close",
  "command": "ClosePosition",
  "payload": {
    "ticket": 58923412,
    "symbol": "EURUSD",
    "volume": null          // null for full position closure
  },
  "error": null,
  "version": "1.0"
}
```

#### 4.2.2. Successful Response Payload

```json
{
  "messageType": "Response",
  "requestId": "req-999-close",
  "command": "ClosePosition",
  "payload": {
    "success": true,
    "ticket": 58923412,
    "brokerMessage": "Position closed successfully."
  },
  "error": null,
  "version": "1.0"
}
```

## 5. MQL5 – `NexusBridge.mq5` Handler Responsibilities

1. **JSON Property Extraction**: Features robust extraction helper methods (`GetJsonStringValue`, `GetJsonDoubleValue`, and `GetJsonIntValue`) capable of reading both quoted strings and unquoted numeric values inside payload trees.
2. **Trade API Validation**: Assures essential properties (Symbol, Volume, and Type) are fully populated and selectable on-terminal.
3. **MqlTradeRequest Execution**: Prepares trading requests matching MT5 execution filling policies (e.g. `ORDER_FILLING_FOK` / `TRADE_ACTION_DEAL`) and dispatches via native `OrderSend()`.
4. **Structured Error Dispatching**: Maps terminal rejections to standardized bridge error codes (such as `TRADE_REJECTED`, `INVALID_PAYLOAD`, and `POSITION_NOT_FOUND`).
5. **Escape Serialization Safety**: Safely escapes string outputs to prevent malformed or broken JSON transfers.

## 6. Sequence Flows

### 6.1. Place Order Sequence

```text
[Nexus Brain C#]                 [TcpBridge]                 [NexusBridge EA]
       |                              |                             |
       |-- PlaceOrderRequest ------>  |-- Forward request ------->  |
       |   (requestId: "req-123")     |                             |-- Validate & OrderSend()
       |                              |                             |-- Query trade result
       |                              |  <-- PlaceOrderResponse ----|
       |  <-- Correlate & Deserialize-|      (ticket: 58923412)     |
```

### 6.2. Position Synchronization Sequence

```text
[Nexus Brain C#]                 [TcpBridge]                 [NexusBridge EA]
       |                              |                             |
       |-- GetOpenPositionsRequest -> |-- Forward request ------->  |
       |                              |                             |-- Enumerate active positions
       |                              |                             |-- Format list to JSON Array
       |                              |  <-- Positions JSON Array --|
       |  <-- Map & Update UI --------|                             |
```

## 7. Mode Selection – Simulated vs Real Bridge

The C# engine supports dual-mode execution:

### 7.1. Configuration
```json
{
  "SelectedProvider": "PostgreSQL",
  "Mt5Mode": "Real",
  "Mt5BridgeHost": "127.0.0.1",
  "Mt5BridgePort": 5000,
  "Mt5BridgeUseSsl": false
}
```

### 7.2. Routing
DI registrations dynamically load `RoutingMt5ConnectionService`, `RoutingMt5AccountService`, and `RoutingMt5TradeService`.
Calls are automatically delegated to the active adapter:
- `SimulatedMt5ConnectionService` / `SimulatedMt5AccountService` / `SimulatedMt5TradeService` (for offline dev/evaluation).
- `RealMt5BridgeConnectionService` / `RealMt5BridgeAdapter` (for real bridge execution).
