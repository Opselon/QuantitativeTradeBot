# ADR-007: Automated Execution Sandbox & Risk-Controlled Runtime Architecture

## Status
Approved

## Context
As the Nexus Trading Engine (NTE) matures toward production-readiness, we require a professional, secure, and resilient execution subsystem. This subsystem must route automated decision signals generated by strategy hosts or AI intelligence modules to execution gateways.
Crucially, the execution layer must satisfy the following constraints:
1. **Broker & MT5 Independence**: The application domain and pipeline must remain completely agnostic of the physical broker platform or MetaTrader 5 terminal interfaces.
2. **Simulation-First Safeties**: Safe paper-trading simulation must be supported natively without needing active terminal/broker connections.
3. **Airtight Risk Controls**: No trade decision from AI or any strategy may route to a broker without passing multiple rigid pre-trade risk validations (Decision -> Risk Validation -> Execution Permission -> Routing).

## Decisions
We introduce the `Nexus.Execution` project/namespace to encapsulate all order lifecycle tracking, state machines, and execution gateways.

### 1. Gateway Abstraction & Adaptors
We define **`IExecutionGateway`** with the following platform-agnostic methods:
- `SubmitOrderAsync`
- `ClosePositionAsync`
- `ModifyPositionAsync`
- `GetPositionsAsync`

We implement two distinct adapter gateways:
- **`SimulationExecutionGateway`**: Executes trading decisions virtually in-memory. Perfect for sandbox simulation, offline paper-trading, and local integration tests.
- **`MT5ExecutionGateway`**: Adapts the MetaTrader 5 bridge client by wrapping `IMt5TradingService` to format payloads and forward trades to `NexusBridge.mq5`.

### 2. The Strict 4-Step Pipeline Flow
Every transaction request is processed via **`RiskControlledExecutionEngine`**, which enforces a sequential execution pipeline:
1. **Decision**: An incoming signal is compiled into an immutable `OrderRequest` and tracked as `Created`.
2. **Risk Validation**: The order passes through `IRiskExecutionGuard` which validates multi-dimensional limits (mandatory Stop Loss, Single Position Size limits, Daily Loss caps, Cumulative Exposure bounds, Risk percentage limits, and Market Condition/Regime suitability).
3. **Execution Permission**: The engine checks the active `ExecutionProfile` (Simulation, Paper, Live). If `Live` is selected, it blocks execution unless `IsLivePermissionGranted` has been explicitly enabled.
4. **Order Routing**: The order is translated and dispatched to the appropriate active `IExecutionGateway`.

### 3. State Machine Transitioning
Instead of mutable boolean flags, the order lifecycle transitions through explicit states:
`Created` ──► `Validated` ──► `Submitted` ──► `Accepted` ──► `Filled` / `PartiallyFilled` / `Rejected` / `Closed`

### 4. Traceable Database Audit Logs
Every trade action and validation result is logged and persisted via `IExecutionAuditService` mapped to `NexusDbContext` tables:
- `orders`
- `positions`
- `trades`
- `execution_errors` (A new dedicated schema for persisting error codes and gateway failures)

## Consequences
- **Absolute Decoupling**: MT5-specific data formats are isolated inside `MT5ExecutionGateway`, preventing leakage of broker concepts into application cores.
- **Flawless Safety**: Risk evaluation is a mandatory, non-bypassable pre-requisite.
- **Simplified Testing**: Complete execution cycles can be fully validated in headless environments using `SimulationExecutionGateway` with zero live network dependencies.
- **Experience Ingestion Readiness**: Decoupled events (`OrderSubmittedEvent`, `OrderFilledEvent`, `OrderRejectedEvent`, `PositionClosedEvent`) are exposed under `Nexus.Execution/Events`, allowing future learning and experience modules to consume audit details without direct dependency loops.
