# Autonomous Learning & Training Engine Documentation

The **Nexus Training Engine** (`Nexus.Training`) provides the foundational self-improving feedback loop for the platform, enabling offline model training, multi-gate validation, and model life cycle management.

---

## 1. Learning Lifecycle

The learning lifecycle follows a structured sequence that ensures the system learns from validated experience rather than reacting blindly to individual live trades:

```text
  Live/Simulated Trading
            │
            ▼
   Experience Collection (ExperienceEngine)
            │
            ▼
    Experience Replay (ExperienceReplayBuffer)
            │
            ▼
   Multi-Gate Validation (ValidationEngine)
            │
            ▼
    Model Promotion (ModelRegistry)
```

1. **Experience Collection**: Trading decisions are recorded as initial `ExperienceSample` instances. Once a trade exits, the physical results (ExitPrice, MaxDrawdown, HoldingTime, Result) are compiled.
2. **Replay Buffer Ingestion**: Experiences are stored in a thread-safe `ExperienceReplayBuffer` supporting various sampling strategies to break temporal dependencies.
3. **Training & Validation**: The offline pipeline trains model candidates and executes strict statistical checks before any model can be promoted.

---

## 2. Quantitative Data Flow

The training pipeline uses an offline-first design to prepare and validate datasets:

```text
  Replay Buffer
       │
       ▼
  [Data Validation] ──(Rejects malformed features or invalid prices)
       │
       ▼
  [Feature Preparation] ──(Scales/normalizes raw 64-element vectors)
       │
       ▼
  [Dataset Creation] ──(Splits into 75% Train / 25% Validation)
       │
       ▼
  [Training Optimizer] ──(Mock gradient descent updates model weights)
       │
       ▼
  [Validation Engine] ──(Runs Backtest, Walk-Forward, OOS, Paper Gates)
       │
       ▼
  [Model Registry] ──(Saves artifacts and promotes to Active)
```

---

## 3. Model Lifecycle & Registry

Models in the system transit through explicit statuses to guarantee live execution safety:

```text
  [Experimental] ──► [Testing] ──► [Approved] ──► [Active]
                                       │              │
                                       ▼              ▼
                                  [Rejected]     [Deprecated]
```

- **Experimental**: Newly compiled or partially trained models.
- **Testing**: Models currently undergoing multi-gate validation.
- **Approved**: Models that successfully passed all 4 validation gates.
- **Active**: The single selected model running live or in paper-simulation routing engines.
- **Rejected**: Models failing to pass validation gates.

---

## 4. Timeframe Learning Separation

To prevent conflicting behavioral rules, training is strictly divided into three learning paths:

| Category | Timeframes | Target Characteristics |
| :--- | :--- | :--- |
| **Scalping** | M1, M5, M15 | High-speed flow, ultra-tight stops, low holding time |
| **Intraday** | M30, H1 | Mid-range volatility, session momentum |
| **Swing** | H4, D1 | Macro trends, wide stops, long holding periods |

Each learning path manages its own datasets, models, and performance metrics independently.

---

## 5. Future AI Integration Plan

As the platform moves to production-grade deep learning, the foundation established in Phase 06 will integrate directly with ML frameworks:

1. **PyTorch / TensorFlow Bridge**: The dataset generated by `TimeframeLearningManager` will be exported to HDF5 or CSV files for direct ingestion by deep neural network models in Python.
2. **ONNX Compiler Pipeline**: Trained network weights will be exported to ONNX format.
3. **Automated Loading**: The `FileModelStorage` artifacts will be read by `NeuralModelService` via `Microsoft.ML.OnnxRuntime` on the live tick hot path, closing the self-improving loop autonomously.
