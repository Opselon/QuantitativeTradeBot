# Nexus Trading Engine - Database Foundation & Abstractions

This document describes the persistence strategy, relational schemas, database migration patterns, optimization models, and cross-provider configuration abstractions inside the Nexus Trading Engine.

---

## 1. Dual-Provider Architecture

The database infrastructure supports a seamless dual-mode model to facilitate both low-friction development and high-throughput production deployment:

```text
                           ┌──────────────────────────────┐
                           │      IUnitOfWork / Repo      │
                           └──────────────┬───────────────┘
                                          │
                                          ▼
                           ┌──────────────────────────────┐
                           │      NexusDbContext (EF)     │
                           └──────────────┬───────────────┘
                                          │
                  ┌───────────────────────┴───────────────────────┐
                  ▼                                               ▼
     ┌─────────────────────────┐                     ┌─────────────────────────┐
     │      SQLite Provider    │                     │   PostgreSQL Provider   │
     │   (Local Dev / Testing)  │                     │  (Production/Enterprise)│
     └─────────────────────────┘                     └─────────────────────────┘
```

1. **SQLite (Development & Integration Testing)**:
   * **Purpose**: Provides a zero-dependency, local-file SQL instance that is initialized instantly via `EnsureCreated()`.
   * **Usage**: Powering local workstations, fast onboarding, and offline scenario simulation tests.
2. **PostgreSQL (Production Enterprise Workloads)**:
   * **Purpose**: Delivers a highly optimized, concurrently accessible, partitioned, and scalable production state store.
   * **Usage**: Runs transactional orders, account states, and records high-frequency analytical results.

---

## 2. Structural Abstractions & Ports

Decoupling is achieved by enforcing interfaces inside the application and core layers:

* **`IUnitOfWork`**: Orchestrates transactional boundaries across multiple repositories and ensures atomic writes via `SaveChangesAsync()`.
* **Repositories (`IAccountRepository`, `IOrderRepository`, `IPositionRepository`, `IMarketDataRepository`)**: Standard abstractions protecting domain entities from database models.
* **`IDatabaseBootstrapper`**: Abstract provider configuration bootstrapping runtime connections (`SqliteDatabaseBootstrapper` vs. `PostgreSqlDatabaseBootstrapper`).

## 3. High-Throughput Database Optimizations (PostgreSQL)

To handle the immense data throughput generated by high-frequency market tick streams, the platform leverages raw, highly specialized PostgreSQL DDL structures located under `src/Nexus.Infrastructure/Persistence/Scripts/`:

### A. Raw Schema and Table Design (`001_create_schema.sql`)
Declares normalized tables for trading state (`Accounts`, `Orders`, `Positions`, `Trades`) and quantitative logging data (`ExperienceRecords`).

### B. Monthly Partitioning Strategy (`002_create_market_partitions.sql`)
To prevent standard search performance degradation as database sizes exceed millions of rows, high-throughput tables (`ticks` and `bars`) are **partitioned monthly** by their native timestamps:
```sql
CREATE TABLE market_ticks (
    id UUID NOT NULL,
    timestamp TIMESTAMPTZ NOT NULL,
    symbol VARCHAR(32) NOT NULL,
    bid DOUBLE PRECISION NOT NULL,
    ask DOUBLE PRECISION NOT NULL,
    volume DOUBLE PRECISION NOT NULL,
    PRIMARY KEY (id, timestamp)
) PARTITION BY RANGE (timestamp);
```
During boot, range-based partitions are dynamically appended to isolate monthly chunks, allowing efficient storage tiering and rapid data pruning.

### C. Specialized Market Indexing (`003_create_indexes.sql`)
Deploys index structures to accelerate search traversals:
* **Composite B-Tree Indexes**: Created over `(symbol, timestamp DESC)` for fast slice-queries fetching recent market windows.
* **Partial Indexes**: Built to target outstanding orders or open positions to speed up core broker simulation operations.

## 4. Optimistic Concurrency Engine

Trading platforms require strict protection against race conditions (e.g., executing two close orders on the same position simultaneously). The platform enforces optimistic concurrency.

### The PostgreSQL `xmin` Column Version mapping
PostgreSQL provides a system column named `xmin` which stores the transaction ID of the last modifying transaction. EF Core is configured to bind this system token in PostgreSQL environments:

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    base.OnModelCreating(modelBuilder);

    if (Database.IsNpgsql())
    {
        // Bind Postgres xmin system column as the active concurrency token
        modelBuilder.Entity<AccountDbModel>()
            .Property<uint>("xmin")
            .HasColumnName("xmin")
            .IsRowVersion();
    }
}
```

### SQLite Fallback Token
Since SQLite does not possess a system transaction ID column like `xmin`, the persistence engine automatically falls back to an mapped software-level concurrency property when running on SQLite, maintaining unified API behavior across providers.

## 5. Temporal Policies & Strict UTC Enforcement

To prevent local vs. server timezone bugs that corrupt chronological market records, the platform enforces a **Strict UTC Date Policy**:

1. **Kind Normalization**: Any `DateTime` read from or written to the database with `DateTimeKind.Unspecified` is automatically treated as `DateTimeKind.Utc`.
2. **Runtime Protection**: The `NexusDbContext` intercepts save calls, auditing all entities. If a property of type `DateTime` with `DateTimeKind.Local` is encountered, it throws an `InvalidDateTimeKindException` to protect data integrity:
   ```csharp
   if (property.CurrentValue is DateTime dt && dt.Kind == DateTimeKind.Local)
   {
       throw new InvalidTimeZoneException("Entities cannot be persisted with local DateTime fields. Only UTC is allowed.");
   }
   ```
3. **UTC Value Converters**: Automatically attached to all datetime properties in Entity configuration mappings.
